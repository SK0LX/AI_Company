# ТЗ — AI Office v3: автономные агенты на подписке (переписывание оркестрации)

Статус: черновик к реализации. Заменяет CEO-push-оркестрацию на **pull-модель**:
дёшево решить «моё / не моё» → адресно сделать → атомарно захватить задачу.
Движок — Claude под подпиской пользователя (фикс-стоимость вместо токенов).

---

## 1. Цель и в чём суть
1. На сервере крутится Claude, **авторизованный подпиской пользователя** (Max), а не API-ключами → стоимость фиксированная.
2. Пользователь пишет сообщение → оно уходит в систему → **поднимаются агенты**.
3. Каждый агент **дёшево** решает «это моя зона?» — **большинство молчит**.
4. Релевантный агент **сам думает, как ответить/сделать**, и постит результат через **reply-tool**.
5. Перед правкой проекта агент **атомарно захватывает задачу/ресурс** (никакой двойной работы).
6. Агент видит **текущие лимиты/план/бюджет** и сам себя тормозит.

## 2. Принципы (инварианты системы)
- **И1. Дёшево решить, дорого сделать.** Relevance-gate стоит ≪ работы; реальная работа — только у релевантных.
- **И2. Атомарный захват.** Нет работы над общим ресурсом без успешного claim (compare-and-set). Двойная работа невозможна by design, а не «по договорённости».
- **И3. Подписка как бюджет.** Окна/лимиты подписки видимы агентам; hard-stop при превышении.
- **И4. Изоляция.** Агент = свой процесс/контейнер; bash — в песочнице (контейнер = граница безопасности, ручной аппрув не нужен).
- **И5. Координатор, не диктатор.** CEO/Router = лок-сервер + арбитр конфликтов + финальный синтез, НЕ пошаговый планировщик.
- **И6. Анти-луп.** Раунд агентов запускает только человеческое сообщение; межагентные цепочки — с жёстким капом глубины.
- **И7. Всё событийно и аудируемо** (доска, таймлайн, активность, cost-ledger).

## 3. Архитектура
```
            Telegram / Android / Web
                     │  (сообщение)
              ┌──────▼──────┐
              │  GATEWAY /   │  приём, история, анти-луп, отдача ответов (стрим)
              │   ROUTER     │
              └──────┬──────┘
            публикует IncomingMessage в ШИНУ
   ┌───────────┬─────┴─────┬───────────┐   (всем агентам)
┌──▼──┐     ┌──▼──┐     ┌──▼──┐     ┌──▼──┐
│Agent│ ... │Agent│     │Agent│     │ CEO │   каждый: relevance-gate → (claim) → work → reply
│ W1  │     │ W2  │     │ Wn  │     │coord│
└──┬──┘     └──┬──┘     └──┬──┘     └──┬──┘
   └───────────┴── claim/release / events ──┴──────┐
                          ┌──────────────────────────▼─┐
                          │  COORDINATION / BOARD       │  task + resource_lock (атомарный CAS),
                          │  + PERSISTENCE + BUDGET      │  владение, статусы, бюджеты, активность
                          └─────────────────────────────┘
```
Компоненты:
- **Gateway/Router** — единая точка приёма/отдачи (Telegram-слой уже есть), анти-луп турбёрста, стрим ответов наружу.
- **Agent Worker** (×N) — процесс/контейнер: Claude-сессия + role-промпт + tools + бюджет.
- **Coordination/Board** — задачи + `resource_lock` с атомарным захватом; «кто над чем».
- **Message Bus** — доставка сообщений агентам, межагентные сообщения, claim/release-события (расширить существующий `bus`/`events`).
- **Budget/Settings** — лимиты/план/остаток квоты агентам; hard-stop.
- **Persistence** — SQLite (→ Postgres при росте).

## 4. Runtime: «Claude на сервере» — ПРОВЕРЕНО (Ф0)
Результаты ресёрча по официальной документации Anthropic (2026-06):
- **Headless — да.** `claude -p` (print mode) и **Claude Agent SDK** (`query()`) — официальные способы гонять агентов программно/без терминала. [docs](https://code.claude.com/docs/en/headless), [Agent SDK](https://code.claude.com/docs/en/agent-sdk/overview)
- **🚨 Авторизация: для серверного/автоматизированного использования НУЖЕН API-ключ, а НЕ подписка.** Подписочный OAuth (`/login`) — только для интерактивного CLI; **Consumer Terms запрещают** использовать подписочные OAuth-токены вне официального CLI. Anthropic прямо: «разработчики продуктов на Agent SDK должны использовать API-key auth». [auth](https://code.claude.com/docs/en/authentication), [SDK+plan](https://support.claude.com/en/articles/15036540) → **идея «авторизуемся под собой ради фикс-цены» для серверного мультиагента НЕ проходит по ToS.**
- **Мультиагент разрешён** — но на **API-ключе** (по-токенно, один ключ питает много агентов; нет per-seat). [usage policy](https://support.claude.com/en/articles/12005017)
- **Конкурентность:** жёсткого лимита нет; практично ~3–6 параллельно; лимиты Max (5-часовое окно + недельный потолок) — **общие** на весь аккаунт.
- **Стоимость (реальность):** 6 агентов 24/7 ≈ **~$2.3K/мес** на API по стандартным ставкам (Sonnet $3/$15 за 1M). Max 20x ($200) тянет реалистично ~4 агента до троттла. На масштабе **API дешевле** подписки + том-дискаунты/Batch −50%/кэш −90%.

**Вывод по рантайму:** движок = **Claude Agent SDK на API-ключе** (по-токенно), стоимость душим нашими **бюджетами с hard-stop** (готово) + **relevance-gate** (готово) + кэш/Batch. Фикс-цена подписки для этого кейса невозможна — это и есть ответ на вопрос про ключи.

- Каждый Agent Worker = одна сессия SDK c: системным промптом роли, белым списком tools, рабочей директорией, бюджетом, доступом к доске.
- Docker: образ Agent Worker (Claude SDK + tools + bash-песочница), параметризован ролью; общий volume для проектных репозиториев (доступ — через claim + git-worktree).

## 5. Поток обработки сообщения (по шагам)
1. **Приём.** Gateway пишет сообщение в историю/чат, публикует `IncomingMessage{chat, text, ts}` в шину, открывает «турбёрст» (И6).
2. **Relevance-gate (дёшево).** Каждый агент получает сообщение и решает «моё / не моё» (см. §6). Большинство → `silent`.
3. **Work.** Прошедшие gate:
   - 3a. Если работа трогает общий ресурс/проект → **claim** задачи/ресурса (см. §7). Не захватил → вежливый ответ «занят X, подключусь после» + подписка на release.
   - 3b. Делает работу (bash/файлы/ресёрч) в своей песочнице, сверяясь с `budget.remaining()`.
   - 3c. **reply-tool**: постит ответ/результат наружу + событие на доску/таймлайн.
4. **Синтез (опц.).** Если ответили несколько — CEO/Router делает лёгкий дедуп/синтез перед отдачей, либо отдаём стримом по мере готовности.
5. **Аудит.** Все шаги → активность/таймлайн/cost-ledger.

Последовательность claim (anti-race):
```
agent → board.claim(resource) → CAS UPDATE → rowcount?
   1 → захватил → работает → board.release()
   0 → занят → board.who(resource) → reply("занят {owner}") + подписка на release
```

## 6. Relevance-gate («моё / не моё», дёшево)
Цель: отсеять нерелевантных без затрат.
- **Шаг 1 — эвристика (0 токенов):** адресность (имя/роль агента в тексте), ключевые слова зоны роли, явный work-intent, приветствие. → если однозначно да/нет — решаем без LLM. (Переиспользуем `group_chat.detect_addressed/work_intent/is_question`.)
- **Шаг 2 — дёшевый LLM (только при неоднозначности):** мелкая/быстрая модель, промпт «ответь yes/no(+3 слова почему): это твоя зона?». Бюджет gate ≪ работы.
- **Анти-шум:** агент видит доску активных claim'ов; если задачу уже взяли — молчит.
- **Конфиг:** пороги, «всегда смотрящие» роли (CEO), cooldown, ambient-вероятность.
- База: текущий `agroup_decide` — это и есть прото-relevance-gate; апгрейдим до двухшагового (эвристика → дешёвый LLM) и добавляем reply/act-tool.

## 7. Атомарный захват задачи/ресурса (claim/lock) — критично
Проблема: два агента одновременно спрашивают «свободно?», оба слышат «да» → двойная работа.
Решение — **compare-and-set в БД (атомарно)**:
- Расширить `task`: `claimed_by`, `claimed_at`, `lock_token`, (есть `status`).
- `claim(task_id, agent)`:
  ```sql
  UPDATE task
     SET claimed_by=:agent, claimed_at=:now, status='in_progress', lock_token=:tok
   WHERE id=:task_id AND (claimed_by IS NULL OR status IN ('new','review'))
  ```
  Проверяем `rowcount`: `1` = захватил, `0` = опоздал. Одна транзакция → гонки нет.
- Для **области/ресурса** (не конкретной задачи) — таблица `resource_lock(key UNIQUE, agent, token, expires_at)`; захват = `INSERT ... ON CONFLICT DO NOTHING` или CAS по `expires_at < now`. Ключи: `repo:project-x`, `file:src/app.py`, `area:frontend`.
- **TTL + heartbeat:** упавший агент не держит лок вечно — `expires_at`, агент шлёт heartbeat; протух → авто-release.
- **release** по завершении + авто-release по TTL.
- «Спросить CEO, не занят ли кто» = `SELECT` для UX-ответа «занят X»; ИСТИНА — атомарный claim, а не select.
- **Очередь/блокеры:** занято → завести blocker (есть `HelpRequest`/blockers) или подписаться на release-событие и продолжить, когда освободится.

## 8. Инструменты агента (tools)
| Tool | Назначение |
|---|---|
| `reply(text)` | ответить пользователю/в чат (reply-tool) |
| `board_query(area\|task)` | кто над чем работает |
| `board_claim(task\|resource)` / `board_release(...)` | атомарный захват/освобождение |
| `board_create_task` / `board_set_status` | лайфцикл задачи (есть) |
| `ask_ceo(question)` | спросить координатора (конфликт/маршрут) |
| `bash(cmd)` | в песочнице контейнера (без ручного аппрува) |
| `files_*` / `git_*` | правка кода; общий репо — через claim + git-worktree |
| `budget_remaining()` | остаток квоты/плана для self-throttle |
| `memory_*` | общая вики/память (есть) |

## 9. Координация и анти-конфликт
- Правка общего проекта: `claim(resource)` → **git-worktree на агента** (параллельные правки без затирания) → merge через CEO/ревьюера.
- CEO = лок-координатор + арбитр + финальный ревью/синтез (не планировщик).
- Анти-луп (И6): раунд запускает только человек; межагентные сообщения — с капом глубины (есть в group-burst).
- Дедлоки/голодание: TTL на локи, очередь с fairness, лимит ожидания → эскалация CEO.

## 10. Бюджеты / лимиты / план (self-regulation)
- В контекст агента: текущий план (Max/…), окна/остаток, per-agent бюджет (cost-ledger есть).
- Перед дорогой работой — сверка с `budget_remaining()`; near-limit → отложить/сообщить.
- Hard-stop (есть): глобально и на агента.
- Трекинг **подписочных окон** (5ч/неделя) — отдельный счётчик использования, чтобы relevance-gate душил лишнее ДО упора в лимит.

## 13. Миграция (что оставить / переписать)
- **Оставить:** реестр, доска/задачи/события, бюджеты, approvals, активность, дашборд, Telegram, group-decide.
- **Переписать:** оркестрацию `team_graph` (CEO push-delegation → pull + relevance-gate + claim); движок моделей (langchain API → Claude SDK сессии); добавить Agent Worker + Gateway + resource_lock + reply-tool.
- **Безопасно:** старый и новый рантайм за фиче-флагом (`AGENT_RUNTIME=legacy|workers`), переключение поэтапно.

## 14. Фазы реализации
- **Ф0 (ресёрч, до Ф2):** проверить runtime-предположение §4 (подписка/конкурентность/ToS).
- **Ф1 (на текущем движке, быстро):** атомарный claim/`resource_lock` + апгрейд group-decide до relevance-gate v2 + reply/act-tool. Проверяет паттерн «дёшево решить → адресно сделать → claim» без смены движка.
- **Ф2 (пилот рантайма):** один Agent Worker на Claude SDK headless под подпиской.
- **Ф3 (масштаб):** N воркеров в Docker + Gateway + межагентная шина claim/release.
- **Ф4 (прод):** деплой 24/7 + Android/веб-клиент (см. отдельное ТЗ android-app-design).

## 15. Критерии приёмки (на фазу)
- Ф1: два агента не могут взять одну задачу (claim атомарен, тест на гонку); нерелевантные молчат; релевантный отвечает через reply-tool; всё на доске/в активности.
- Ф2: один воркер на подписке отвечает на сообщение end-to-end, расход идёт по подписке, бюджет виден.
- Ф3: N воркеров параллельно, без двойной работы и без затирания файлов (worktree), Gateway стримит ответы.

## 16. Риски / открытые вопросы
- **ПРОВЕРИТЬ:** подписка на сервере, конкурентность сессий, ToS (load-bearing — §4/Ф0).
- Дедлоки/голодание на claim → TTL, очередь, fairness, эскалация.
- Стоимость даже на подписке (окна) → relevance-gate критичен.
- Мерж параллельных правок → worktree + обязательный ревью.
- Безопасность bash → контейнер-песочница + денайлисты (.git/.env/секреты — уже есть в self-modify).
- Сложность межагентной шины → начинать с минимума (claim/release + reply), не строить лишнего.
